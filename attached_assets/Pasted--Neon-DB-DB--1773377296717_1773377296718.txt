현재 문제는 코드 수정 후 자동 재배포가 일어날 때 서버가 재시작되면서 Neon DB 재연결이 실패하면, 그 이후 요청들이 실제 DB 오류임에도 불구하고 빈 데이터처럼 반환되는 현상입니다.

이 문제를 “임시 재시도” 수준이 아니라 구조적으로 해결해 주세요.

반드시 아래 요구사항을 모두 반영해 주세요.

[핵심 목표]
1. DB 연결 실패를 빈 배열([])이나 빈 객체로 반환하지 말 것
2. DB 장애와 실제 데이터 없음 상태를 명확히 구분할 것
3. 서버 시작 시 DB 연결 확인에 실패하면 서버가 정상 기동되지 않게 할 것
4. DB pool/client를 요청마다 새로 만들지 말고 singleton으로 관리할 것
5. 초기 연결 실패 후 죽은 연결 상태가 고정되지 않게 할 것
6. 실패 원인을 로그로 명확히 남길 것

[구체 작업 지시]

1) 프로젝트 전체에서 DB 관련 코드 검색
- catch 구문에서 return [] / return {} / return null 로 DB 오류를 숨기는 부분을 모두 찾기
- 특히 “조회 실패 시 빈 배열 반환” 패턴이 있으면 제거하기
- DB 쿼리 실패는 반드시 throw 하거나 상위 레이어로 전달하기

2) DB 연결 구조 개선
- Neon/Postgres 연결 pool을 전역 singleton으로 관리하기
- 요청마다 new Pool(), new Client(), drizzle(db), neon() 등을 새로 만들고 있다면 수정하기
- 앱 프로세스 내에서 재사용 가능한 단일 DB 인스턴스 구조로 바꾸기
- pool max, idle timeout, connection timeout을 명시적으로 설정하기

3) startup health check 추가
- 서버 시작 전에 반드시 간단한 DB health check 실행: SELECT 1
- health check는 5회 재시도, 재시도 간격 2초
- health check가 끝까지 실패하면 서버 listen 하지 말고 프로세스를 종료하거나 명확히 실패 처리하기
- 즉, DB가 죽어 있는데 앱만 정상처럼 뜨지 않게 하기

4) API 에러 처리 수정
- DB 장애 발생 시 API는 200 + 빈 배열을 반환하지 말 것
- DB 연결/쿼리 실패 시 503 status와 명확한 error code를 반환할 것
- 예: { ok: false, error: "DATABASE_UNAVAILABLE" }
- 실제 데이터가 없는 경우에만 200 + 빈 배열 또는 정상 응답 반환

5) 재시도 정책 정리
- 초기 연결 확인(boot time)에는 재시도 허용
- 일반 요청 중 DB 조회 실패는 무한 재시도하지 말 것
- 읽기 쿼리는 필요하면 제한적 재시도 가능하지만, 쓰기 작업(insert/update/delete)은 중복 실행 위험이 있으니 자동 재시도에 주의할 것

6) 로그 강화
- 서버 시작 시 DB health check 성공/실패 로그 남기기
- DB query 실패 시 route, error message, timestamp 남기기
- “빈 데이터”와 “DB 장애”가 로그상 구분되도록 만들기
- 재배포 직후 장애가 발생했는지 추적 가능하게 로그를 구조화하기

7) 프론트엔드 응답 처리 수정
- 프론트가 빈 배열과 서버 오류를 구분하도록 수정하기
- 503이면 “일시적으로 데이터를 불러올 수 없습니다” 메시지 표시
- 실제 빈 데이터일 때만 “등록된 데이터가 없습니다” 표시

[원하는 결과]
- 코드 수정 후 자동 재배포가 발생해도, DB가 잠깐 불안정하면 startup health check와 재시도로 회복
- 그래도 DB 연결이 안 되면 앱이 가짜 정상 상태로 뜨지 않음
- DB 오류가 발생해도 사용자가 “데이터가 사라졌다”고 보지 않음
- 실제 DB 장애는 503으로 노출되고 로그에 정확히 기록됨
