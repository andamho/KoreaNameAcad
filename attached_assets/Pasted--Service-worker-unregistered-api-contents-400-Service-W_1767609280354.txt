“Service worker unregistered”가 찍혔고, 이제도 /api/contents가 400으로 떨어지는 걸 보면

✅ Service Worker 문제는 정리됐고
❌ 이제 남은 건 서버(프로덕션)에서 DB init이 여전히 실패하고 있다는 뜻입니다.

즉, 다음 단계는 “왜 initDatabase() 테스트 쿼리가 실패하는지”를 서버 로그로 잡아내는 것이에요. 지금 단계에선 이게 가장 빠르고 확실합니다.

1) 지금 콘솔에서 확정된 사실

SW 제거 완료 ✅

/api/contents 호출은 서버까지 감 ✅

서버가 400 반환 ❌

이 400의 본문(이전처럼)도 다시 확인해야 합니다.
Network에서 /api/contents 클릭 → Response 탭을 보세요.

👉 만약 아직도

{"details":"Database not available"}


면 DB 연결 실패가 그대로고,

👉 만약 다른 validation 에러로 바뀌었으면
(예: title/content 길이 등) 그건 다음 단계입니다.

2) 지금 바로 해야 할 3가지 (Replit 기준)
✅ A. Replit “프로덕션(Deployments)” 로그에서 DB init 에러 확인

Replit에서 배포가 Deployments로 되어 있으면:

Replit 왼쪽 메뉴 → Deployments

현재 배포(Production) 클릭

Logs 탭에서 DB init failed 또는 postgres 관련 에러 찾기

📌 그런데 지금 initDatabase() 코드가 에러를 console.error로 찍지 않으면 로그에 아무것도 안 나옵니다.

그래서 반드시 아래처럼 찍어야 합니다.

server/storage.ts 수정(필수)
private async initDatabase() {
  try {
    const { db } = await import("./db");
    await db.select().from(nameStories).limit(1);
    this.dbAvailable = true;
    console.log("✅ DB init ok");
  } catch (err) {
    console.error("❌ DB init failed:", err);
    this.dbAvailable = false;
  }
}


이걸 배포한 뒤 Logs 보면,

SSL 문제인지

연결문자열 문제인지

권한/테이블 문제인지
한 방에 나옵니다.

✅ B. server/db.ts에서 Neon SSL 강제 적용이 “프로덕션에 실제 반영”됐는지 재확인

아래가 배포 코드에 들어갔는지가 핵심입니다.

const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  ssl: { rejectUnauthorized: false },
});


⚠️ Replit에서 흔한 실수:

로컬(Repl Run)엔 반영됐는데

Deployments는 이전 빌드라서 반영 안 됨

→ Deployments에서 Redeploy가 필요할 수 있어요.

✅ C. DATABASE_URL 자체가 프로덕션에서 “진짜 존재”하는지 10초 체크

서버 시작 시 한 줄만 찍어도 됩니다 (값 전체 말고 존재 여부만)

console.log("DATABASE_URL exists?", !!process.env.DATABASE_URL);


만약 false면 → “자동 설정됐다”는 가정이 틀린 거고, Deployments에 env가 안 들어간 겁니다.

3) 바로 해결될 가능성이 높은 원인 TOP 3 (Neon+Replit)

로그에서 가장 자주 나오는 순서:

SSL required / self-signed certificate / no pg_hba.conf entry
→ ssl: { rejectUnauthorized:false }로 해결

timeout / ETIMEDOUT
→ Neon 프로젝트 sleep / 네트워크 / region 문제

relation "nameStories" does not exist
→ 마이그레이션/스키마가 프로덕션 DB에 없음
(dev/prod 같은 DB 공유라면 덜하지만, 테이블명이 실제 DB와 다른 경우도 많아요)
